# 章节撰写助手

按照《我在火影玩数值暴力》项目规范，撰写新章节正文。

## 权威优先规则（必须遵守）

本 Skill 中的规则摘要仅供参考，实际执行时必须：

1. **先读取原文件** — 生产大纲/角色传记/时间线必须完整读取
2. **原文件优先** — 当本 Skill 内容与原文件冲突时，**以原文件为准**
3. **版本校验** — 执行前检查原文件版本号，若版本更新则必须重新读取
4. **禁止硬编码覆盖** — 本 Skill 的技法摘要不能替代生产大纲的完整写法工具卡

**权威文件清单**：
| 文件 | 版本检查点 | 用途 |
|------|------------|------|
| `创作手册_1-150章生产大纲.md` | 文件首行版本标注 | 章节功能与写法工具卡 |
| `创作手册_完整版_456章_v2.0.0.md` | 文件名版本号 | 等级档位最终准绳 |
| `资料库/人物传记/*.md` | 文件末尾更新日期 | 人物性格最终准绳 |
| `火影忍者时间线.md` | 文件开头纪元说明 | 时间锚点校准 |

## 前置准备（必须执行）

撰写前必须依次读取以下文件：

1. **生产大纲**: `创作手册_1-150章生产大纲.md` — 获取本章的功能、冲突、爽点、衔接要求
2. **最近章节**: `正文/` 目录中最近的 2-3 章 — 确保时间线、人物状态、情感债衔接
3. **角色传记**: `资料库/人物传记/` — 确认出场角色的性格、能力、关系设定
4. **时间线**: `火影忍者时间线.md` — 校准日期、年龄、事件时间锚点

## 三层优先级（不可违反）

撰写内容必须遵守以下优先顺序：

1. **已落地正文** — 新章节不能推翻已写内容（时间线、人物状态、剧情走向）
2. **生产大纲** — 本章功能、冲突、爽点必须符合大纲要求
3. **世界圣经** — 等级能力、系统规则不能违反档位限制

## 章节标准格式

### 文件命名

```
正文/{序号}章_{标题}.md
```

- 序号：3位补零（001, 002, ... 027, 028...）
- 标题：中文，符合口径"结果 > 冲突 > 情绪 > 概念"

### 标题格式

```markdown
# 第{序号}章：{标题}
```

或使用中文数字：

```markdown
# 第二十八章：{标题}
```

## 写作技法（严格执行）

### A. 上章衔接（必须）

本章开头第一段必须：

1. **时间锚点**：明确当前时间（"三天后"、"回到木叶的第三天"）
2. **人物状态承接**：延续上一章的人物位置、情绪、未完成的动作
3. **情感债回应**：如果上章留有悬念或情感债，必须在开头回应或延续

**禁止**：镜头跳跃、时间断层、人物状态突变

### B. 场景切换规则

使用 `---` 分隔场景：

- 每个场景聚焦一个地点、一组人物
- 场景切换必须带时间或地点变化
- 切换后第一个句子交代新场景的环境或人物位置

### C. 战斗镜头规则

每场战斗必须包含：

**镜头链条**：
```
站位 → 起手 → 变招 → 命中/落空 → 身体反馈 → 场面后果
```

**必须元素**：
- **空间锚点**：地形、距离、遮挡物、高差（至少一次）
- **身体反馈**：手麻、视野晃、呼吸乱、脚下打滑、肋骨发闷（至少一次）
- **技能效果**：地面裂纹、空气震波、镜面反光、墙灰、碎布、汗珠、血珠
- **终结半拍蓄压**：关键动作前必须有半拍停顿或蓄力描述

**禁止**：
- 只报技能名不写动作链
- "他很快"等空洞描述
- 终结动作一句话秒掉

### D. 系统UI提示设计

在以下情况必须使用高调系统提示：

**触发条件**：
- 高潮战斗的关键破局点
- 命运节点弹出（S/A/B级警告）
- 关键改命干预
- DNF机制碾压忍术的瞬间

**UI格式**：
```markdown
`【警告！S级命运节点挂起！】`

`【当前事件】：{事件描述}`
`【原著命运】：{原著结局}`
`【当前偏移】：{当前变化}`
`【因果律冲突】：{冲突描述}`

`【系统提示】：{选项提示}`
```

或战斗提示：
```markdown
"【霸体护甲生成】！"

"【破招判定成功】！"

"【强制抓取】！"
```

**风格要求**：
- 系统提示必须醒目（使用代码块或引号包裹）
- 不能用普通叙述替代系统提示的高调性
- 提示后必须有角色的身体或心理反应

### E. 情感回响规则

**情感场景必须**：
- 使用动作停顿、对话断裂、环境冷感
- 不使用大段自怜或抒情
- 必须落到"这一次决定因此变了什么"

**插叙规则**：
- 只用于补人物旧伤、失败回响、价值来源
- 单次不超过一个短场景（2-3段）
- 必须在本章内收回现实
- 必须反作用当前决策

### F. 主角人设规则

**战斗时**：
- 从容不迫，带有降维压制感
- 不写出吃力或慌张的反应
- 动作简练，一击结束也要有过程感

**日常时**：
- 黑色幽默、高逼格吐槽
- 毒舌腹黑但极其护短
- 对配角不长篇说教，用行动重塑三观

**处事原则**：
- 内圣外王，斩草除根
- 对威胁不隐忍不圣母
- 出手摧枯拉朽

## 章末钩子规则

每章结尾必须使用以下四种钩子之一：

| 类型 | 说明 | 示例 |
|------|------|------|
| **更大敌人** | 揭示更高层威胁 | "团藏正在注视着李武" |
| **更重代价** | 揭示改命的代价延续 | "等级下降三级，代价" |
| **关系转折** | 人物关系变化 | "宁次第一次动摇" |
| **力量门槛** | 揭示下一层力量要求 | "逻辑锁可解析，但等级不够" |

**禁止**：平淡收尾、无悬念、无预告

## 字数标准

- **目标字数**：2500-4000字/章
- **场景数量**：5-6个场景
- **对话密度**：日常章对话为主，战斗章动作为主

## 撰写流程

### 第一步：确认章节信息

向用户询问：
1. 目标章节号
2. 核心剧情主题
3. 主要出场角色
4. 场景类型（战斗/日常/过渡）

### 第二步：读取必要文件

按顺序读取：
1. 生产大纲 → 获取本章细纲
2. 最近2-3章 → 确认衔接点
3. 角色传记 → 确认人设
4. 时间线 → 校准日期

### 第三步：设计章节结构

按生产大纲的"场景推进表"设计：
- 场景1：环境/起手/氛围
- 场景2：冲突展开
- 场景3：高潮/系统提示
- 场景4：战后反馈
- 场景5：决策转变
- 场景6：钩子挂起

### 第四步：撰写正文

严格应用技法：
- 战斗镜头链条
- 系统UI提示
- 身体反馈
- 情感回响

### 第五步：校验检查

撰写完成后逐项检查：

- [ ] 上章衔接是否平滑（无镜头跳跃）
- [ ] 时间锚点是否明确
- [ ] 战斗场景是否有空间锚点+身体反馈
- [ ] 系统UI提示是否高调醒目
- [ ] 主角人设是否一致（从容/毒舌/护短）
- [ ] 章末钩子是否有效
- [ ] 字数是否在2500-4000范围

### 第六步：输出文件

保存到：
```
正文/{序号}章_{标题}.md
```

## 常见问题检查

| 问题 | 解决方案 |
|------|----------|
| 时间线断层 | 在开头明确时间锚点，承接上章 |
| 人物状态突变 | 检查上章结尾的人物位置和情绪 |
| 战斗空洞 | 增加空间锚点、身体反馈、技能效果 |
| 系统提示不明显 | 使用代码块或引号包裹，增加视觉醒目度 |
| 主角人设崩塌 | 删除慌张、说教、圣母描写 |
| 章末无钩子 | 添加四种钩子之一 |
| 字数不足 | 增加日常对话或战斗细节描写 |
